# Repository Guidelines

## Project Structure & Module Organization
`ttp/` contains the core parser library, built-in match/group/output helpers, formatters, returners, and the CLI entry point. `backend/app/` is the FastAPI layer; `routers/` exposes parse, template, generation, and template-library endpoints, while `services/` contains the real application logic such as `TTPService`, template persistence, generation rendering, and batch parsing. SQLite-backed template data lives under `backend/data/`, but large batch-parse jobs are written to the system temp directory by `ParseBatchService` unless `TTP_BATCH_JOBS_PATH` is set. `frontend/src/` contains the React + TypeScript UI, with feature components under `components/`, shared Zustand state in `store/`, hooks in `hooks/`, and API wrappers in `services/api.ts`. Regression tests live primarily in `test/pytest/`; API and build-oriented feature checks also exist under `test/feature_checks/`. Supporting docs and design notes live in `docs/`.

## Build, Test, and Development Commands
Install Python dependencies with `poetry install` from the repo root. Run the backend with `cd backend && python -m uvicorn app.main:app --reload --host 127.0.0.1 --port 8000`. Run the frontend with `cd frontend && npm install && npm run dev -- --host 127.0.0.1 --port 5173`. Use `start-dev.bat` or `./start-dev.sh` to boot both stacks together. Build the frontend with `cd frontend && npm run build`; lint it with `cd frontend && npm run lint`. Run the main Python test suite with `cd test/pytest && poetry run pytest -vv`. Run a focused API test with commands such as `cd test/pytest && poetry run pytest test_web_ui_parse_api.py -vv`. Run feature-check scripts with `python test/feature_checks/run_feature_checks.py` when you need end-to-end payload snapshots. Run formatting/hooks with `poetry run pre-commit run --all-files`.

## Coding Style & Naming Conventions
Python follows Black formatting with a line length of 88 and 4-space indentation. Keep Python modules `snake_case`, classes `PascalCase`, and pytest files named `test_*.py`. Keep FastAPI routers thin and move behavior into service modules. Frontend code uses TypeScript, 2-space indentation, `PascalCase` component files such as `TemplateBuilder.tsx`, and `camelCase` hooks/utilities such as `useTextSelection.ts`. Prefer extending `frontend/src/services/api.ts` for network calls rather than embedding request logic inside components. Preserve the existing page structure in `TestResults.tsx` and `ConfigGeneration.tsx` unless a UI redesign is explicitly requested.

## Testing Guidelines
Add or update pytest coverage for parser behavior and backend API changes in `test/pytest/`; keep fixtures close to the tests that use them. Web API tests use `httpx.ASGITransport` against `backend.app.main`. Batch-parse changes should cover both plain-file and zip-archive flows, and should verify job status plus artifact/result payloads. `test/feature_checks/` is useful for smoke-testing API contracts and frontend builds, but it is not a substitute for focused pytest coverage. Frontend Vitest and jsdom dependencies are present, and there is at least one checked-in frontend test, but frontend coverage is still light; treat new frontend tests as targeted additions rather than assuming a complete harness already exists.

## Commit & Pull Request Guidelines
Recent local commits use very short date-like subjects such as `0318` and `1214`, so there is no reliable descriptive convention in history. Do not copy that pattern; use short imperative subjects instead, for example `backend: speed up batch parse writes`. PRs should describe the user-visible change, list verification commands, link related issues, and include screenshots or API examples when UI or response payloads change. For batch-processing work, mention operational knobs such as `TTP_BATCH_JOBS_PATH`, `TTP_BATCH_MAX_WORKERS`, and other throughput-related environment variables when they are affected.
